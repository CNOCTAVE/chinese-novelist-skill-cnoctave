# 第10阶段：（可选）（多用于男频）人物设定 / 情节走向与上文表述冲突优化

**用户通过 AskUserQuestion 选择运行此阶段后执行**
**目标**：检测并消除三类冲突——① 人物设定（`00-人物档案.json`）与正文的矛盾；② 大纲情节走向（`01-大纲.json`）与正文的矛盾；③ 上文与下文（章间 / 段间）的自相矛盾。让人物、剧情、行文三者前后一致，杜绝"人设崩了""前面挖坑后面不填""时间线对不上"等出戏问题。

---

## 何时运行本阶段（两个时机）

本阶段可在两个时机被触发，均由用户**主动选择**是否运行：

- **时机一：用户改完大纲 / 人物设定之后、开始写作之前（Timing 1）**
  用户在第二阶段选择「暂停修改」或直接编辑了 `00-人物档案.json` / `01-大纲.json` 后，于后续对话输入「继续写作」等字样恢复创作。此时用户很可能**漏改了某些内容**——比如改了第 3 章的角色名却忘了同步到第 7 章，或改了人设却没改大纲里的对应关系，导致**章间 / 设定间出现冲突**。本阶段在恢复写作前提供检测，先把冲突掐灭在未写或已写章节里。
- **时机二：skill 已写完小说内容（Timing 2）**
  自动校验（第 4 阶段）通过后，在「第 4.5 步可选增强阶段」中作为选项提供。用于修复 skill 在分章创作过程中**自身产生**的冲突（人物性格前后不一、情节偏离大纲、伏笔未回收、时间线错乱等）。

> 两种时机共用同一套检测 / 呈现 / 修复逻辑，区别仅在**可用材料**：
> - Timing 1 若尚无任何章节写成（仅有人物档案 + 大纲）→ 聚焦「大纲内部一致性」与「大纲 ↔ 人物设定一致性」；若有已写章节 → 一并检测「已写章节 ↔ 新设定」。
> - Timing 2 通常已写完全部章节 → 三类冲突全部检测。

## 0. 运行前准备（确认可用材料）

1. 读取 `02-写作计划.json`，确认 `projectPath`、`status`、章节列表与文件路径。
2. 读取 `00-人物档案.json`（人物设定基准）与 `01-大纲.json`（情节走向基准）。
3. 根据 `status` 与章节 `status` 判断可用材料：
   - 无任何 `completed` 章节 → 仅做**设定间**检测（维度见下「维度四」）。
   - 存在已写章节 → 读取这些章节 `.md` 文件，**三类冲突**全部检测。
4. 更新 `02-写作计划.json` 的 `status` 为 `"conflict_resolving"`（仅当存在已写章节需要修改时；纯设定间检测可跳过状态变更，或同样标记以便追溯）。

> **检测阶段允许通读全部相关材料**（人物档案 + 大纲 + 所有章节），以便发现跨文件 / 跨章节冲突；**修复阶段严格逐文件写回**——一次只改【一个冲突涉及的一处文件】（某一章 `.md` 或 `01-大纲.json` 或 `00-人物档案.json`），禁止一次性批量改写多章多文件。

## 一、冲突检测（仅检测，不修改）

按以下四个维度逐项比对，记录每一处冲突的**位置、双方表述、冲突类型**。

### 维度一：人物设定 ↔ 正文表述

| 冲突类型 | 说明 | 示例 |
|----------|------|------|
| 性格矛盾 | 正文行为与人设 `core_personality` / `fatal_flaw` 相悖 | 人设"沉默寡言、极度谨慎"，正文却让主角当众喋喋不休、鲁莽冒进 |
| 外貌矛盾 | 正文描写与人设 `appearance` 不一致 | 人设"左颊有一道刀疤"，正文写成"右脸的疤"或"脸上光滑无痕" |
| 说话风格矛盾 | 正文对话与人设 `speaking_style` 不符 | 人设"惜字如金、爱用文言"，正文满篇口语废话 |
| 关系矛盾 | 正文人物关系与人设 `relationships` 冲突 | 人设"与王厉是死敌"，正文称其为"挚友" |
| 背景/能力矛盾 | 正文设定与人设 `backstory` / `weakness` 抵触 | 人设"自幼不会半点武功"，正文却大杀四方 |
| 称谓/姓名不一致 | 同一角色前后名字 / 称呼不同 | 前文"林夏"，后文误作"林夕" |

### 维度二：大纲情节走向 ↔ 正文表述

| 冲突类型 | 说明 | 示例 |
|----------|------|------|
| 出场人物不符 | 正文出场人物与大纲 `characters` 不符（多出未规划人物，或漏写规划人物） | 大纲本章出场[甲、乙、丙]，正文凭空多出丁，或丙全程缺席 |
| 核心事件偏离 | 正文写的事件与大纲 `coreEvent` 不一致 | 大纲核心事件是"潜入敌营"，正文写成"街头比武" |
| 场景不符 | 正文场景与大纲 `scenes` 不一致 | 大纲场景[宫廷大殿]，正文在市井酒楼 |
| 关键事件缺失/多余 | 大纲 `keyEvents` 在正文未体现，或正文自添重大事件架空了大纲 | 大纲列 10 个关键词，正文只写了 3 个，且自行加了"主角觉醒异能" |
| 悬念钩子不符 | 正文结尾与大纲 `cliffhanger` 不一致 | 大纲结尾钩子是"信使带来了密信"，正文结尾是"主角安然入睡" |
| 承接断裂 | 本章 `continuationFromPrevious` 回应的是上章钩子 A，但上章实际结尾是 B | 第 2 章结尾"门后传来熟悉脚步声"，第 3 章开头却"第二天阳光明媚，他早已忘记昨夜" |

### 维度三：上文 ↔ 下文（章间 / 段间一致性）

| 冲突类型 | 说明 | 示例 |
|----------|------|------|
| 时间线矛盾 | 前后章节时间先后顺序 / 间隔打架 | 第 3 章"三日后才出发"，第 5 章"昨日刚抵达" |
| 因果倒置 / 丢失 | 伏笔与回收矛盾，或该回收的伏笔没回收 | 第 2 章埋下"玉佩是信物"，第 8 章认亲却用"胎记"，玉佩再无交代 |
| 人物状态前后矛盾 | 人物身体 / 处境前后无交代地突变 | 第 4 章"重伤卧床"，第 6 章生龙活虎无任何痊愈交代 |
| 章间情节矛盾（事件/因果前后打架） | 相邻章的"已发生事实"与"情节前提"直接冲突，无转折交代 | 第1章主角与反派决裂立誓不死不休，第2章却把酒言欢携手合作；或第1章"密信已焚"，第2章"他掏出密信细读" |
| 设定前后矛盾 | 世界观 / 规则前后不一致 | 前文"此国从不信神"，后文主角大搞祭祀祈神 |
| 称谓/身份前后矛盾 | 同一事物 / 身份前后叫法不同 | 前文"镇北将军"，后文变"镇南将军"且无封地变动 |

### 维度四：Timing 1 专属（用户改大纲 / 人设之后）

| 冲突类型 | 说明 | 示例 |
|----------|------|------|
| 大纲内部不一致（章间冲突） | 用户改了某处却漏改关联处；或相邻章大纲的情节前提互相矛盾 | 第 3 章把"苏明月"改为"苏明玉"，第 7、9 章仍称"苏明月" |
| 大纲 ↔ 人物设定不一致 | 大纲引用了人物档案不存在的角色 / 属性 | 大纲出场人物含"影七"，人物档案里根本没有这个人；或大纲说主角"是皇子"，人设写"平民之子" |
| 已写章节 ↔ 新大纲冲突 | 用户改大纲后，前面已写章节按旧设定，与新大纲不符 | 用户把结局从"复仇"改为"和解"，但已写的第 5—10 章全是血腥复仇戏 |

### 章间情节连贯性扫描（相邻章情节冲突专项检测）

"上文 ↔ 下文"与"大纲内部章间"两类冲突，最容易在**相邻两章的情节**上爆雷——前一章写的事，后一章却以相反或矛盾的前提推进。除逐维度比对外，**必须额外做一轮"相邻章情节连贯性扫描"**：

> **先分类：这是「矛盾」还是「疑似新增设定」？** 扫到章间情节差异时，先判断性质：
> - **真矛盾（不可同真）**：前后两章陈述在逻辑上不能同时成立（如第1章"密信已焚"、第2章"他掏出密信细读"；或第1章"主角单身"、第2章"他妻子"且无任何婚/丧事件）。→ 按冲突处理。
> - **疑似新增设定（后章引入新发展，仅缺过渡）**：后章出现的新状态/新元素，前章并未"否决"它，只是没把"怎么变过来"写出来（如第1章主角与反派敌对、第2章却已联手——若中间本应有转折戏却没写，既可能是漏写冲突，也可能是作者想新增"两人暗中联手"的设定但忘了补衔接）。→ 性质存疑，按下方模式分流：自动模式默认当作新增设定、自动补衔接；手动模式交用户选「改冲突」还是「加设定」。

1. **依章节顺序逐对扫描**：对 (第1章,第2章)、(第2章,第3章)…(第N-1章,第N章) 每一对，核对前一章的"已发生事实"与后一章的"情节前提"是否一致。
2. **大纲层扫描**（Timing 1 / 大纲章间冲突）：对每对相邻章，检查
   - 上一章 `cliffhanger`（悬念钩子）与下一章 `continuationFromPrevious`（承接）是否呼应、是否对同一事件说法一致；
   - 下一章 `coreEvent`（核心事件）所依赖的"前文前提"，是否真在上一章（或更早）发生；
   - 相邻章 `characters` / `scenes` / `keyEvents` 衔接是否自然（无凭空出现/消失的人物或事件）。
3. **正文层扫描**（上文 ↔ 下文）：对每对相邻章的正文，检查
   - 前一章结尾确立的**事实状态**（谁活着/死了、结盟/敌对、事件成/败、物件去向）与后一章开头/主线是否一致；
   - 后一章是否"默认了某件前一章并未发生（或已发生相反）的事"；
   - 关键情节走向（如第1章主角决意复仇，第2章却已和解罢手）前后是否矛盾且无交代。
4. 发现矛盾即记入报告，标注「第X章 ↔ 第X+1章」与矛盾点。

> 典型章间情节冲突示例：
> - 第1章结尾"主角与反派当众决裂、立誓不死不休"；第2章开头"两人相视一笑，把酒言欢携手合作"——中间无任何转折，情节直接打架。
> - 第1章大纲 coreEvent"主角接受调令赴边关"；第2章大纲 continuationFromPrevious"回应上章主角拒调留守"——同一件事两章说法相反。
> - 第1章正文"密信已被焚毁"；第2章正文"他掏出那封密信再度细读"——物件状态前后矛盾。

### 姓名 / 称谓冲突的「异名归一」原则（重点避坑）

姓名类冲突最易修错，务必遵守：

1. **以人物档案为唯一真名基准**：凡正文 / 大纲中出现的姓名或称谓，必须在 `00-人物档案.json` 的 `name` 字段中存在；不存在的写法即冲突。
2. **按「身份 / 行为 / 关系」归正，而非按字形相似**：当某处写法与档案里任何人都不一致时，**先用上下文判定它指的是哪一个角色**（谁在做什么、与谁的关系、承担什么功能），再映射到该角色的档案真名。
   - 典型陷阱：**同姓串味**——例如正文把内鬼写成「赵长成」、把 rival 写成「赵长风」，二者字形相近，但「赵长风携吞并协议」里的角色实际是晟世总裁**顾长风**（档案里有、且吞并协议本就是他的动作），不能因为都姓「赵」就错改成内鬼「赵天成」。必须回到「谁干了什么事」来定。
   - 另一陷阱：**称谓性别 / 身份不符**——例如长辈称男主「小琳」（女性化昵称、且非其名），应改为其真名或合理昵称（如「小林」），而非照搬错字。
3. **同人多名也要统一**：若同一角色在正文与大纲里被写成两种写法（如大纲用「赵天成」、正文用「赵长成」），统一为档案真名，避免章间 / 设定间不一致。
4. **改前二次确认**：对「疑似串味」的姓名冲突，若上下文仍无法 100% 判定归属，按 3.1 规则交 AskUserQuestion 让用户拍板，绝不替用户把 A 角色的名字安到 B 角色头上。

## 二、冲突呈现与用户选择（必须执行）

检测完成后，**必须**先向用户呈现报告（格式见上），再用 AskUserQuestion 让用户决定**以何种方式优化**：

Question：检测到上述 [N] 处冲突，选择修复方式？
[A] 自动修复（推荐，省心）→ 按 3.1 权威方判定规则批量处理每一处冲突，全程不打断你
[B] 手动逐条修复（可控）→ 每一处冲突都单独向你确认；当某处有多种改法时，由你选一种作为基准
[C] 仅修复指定类别 → 先从四类维度勾选（multiSelect），再决定这批冲突走自动还是手动
[D] 暂不修复，跳过 → 直接结束本阶段，不做任何修改

> 若选 [C]：先以 AskUserQuestion（multiSelect）列出维度 [维度一 人物设定↔正文] [维度二 大纲情节↔正文] [维度三 上文↔下文] [维度四 大纲/人设内部] 供勾选；用户也可在对话中直接点名"只修第 2、3 处"。勾选完成后，再回到本步询问这批冲突走 [A 自动] 还是 [B 手动]。

> **自动 vs 手动 的区别**：自动模式由 skill 直接套用 3.1 规则落笔；**手动模式在每一处冲突上都停下来，把"可选改法"摆给你选**——满足"每处冲突都提供选项、多种改法由你定基准"的要求。冲突数量很大且高度同质时（如全文同一人名的多种错写），手动模式会逐条询问，你可在任一条选择"剩余同类套用此方案"以加速，但不剥夺逐条选择权。

## 三、冲突修复主循环（可能需要多轮）

> **核心原则：先判定"权威方"，再动手。** 冲突的两边必有一方更可信 / 更代表用户意图，修复时以权威方为准去对齐另一方；无法判定时交回用户决定。
> **运行模式**：本阶段按第二节选定的 [A 自动] 或 [B 手动] 执行。自动模式走 3.2-A；手动模式走 3.2-B（每处冲突都提供选项）。

### 3.1 权威方判定规则（自动模式的默认 / 手动模式的推荐基准）

| 场景 | 权威方 | 动作 |
|------|--------|------|
| Timing 1 用户主动修改过设定 | **用户修改后的大纲 / 人物设定** | 以新设定为准，修正已写正文（或大纲中漏改的关联处） |
| 大纲先于正文生成且未被用户推翻 | **大纲（情节走向）** | 正文偏离大纲 → 通常改正文回归大纲；若偏离明显是更优情节发展，先问用户再定 |
| 人物设定（角色基准） | **人物设定** | 正文与人设冲突 → 改正文；除非用户明确要改人设 |
| 上文 ↔ 下文 | **以后文为准**（后续章节通常更贴近最新设定 / 用户意图） | 修正前文使前后一致；若前文已有强铺垫、改后文成本更低，则调整后文 |
| Timing 2 正文与大纲/人设冲突且难判权威 | **询问用户** | 手动模式下列为候选方案交用户选；自动模式默认以大纲/人设为基准并标注 |
| 章间差异判定为「疑似新增设定」（后章引入新发展，仅缺过渡） | **视运行模式** | 自动模式：当作新增设定，自动补一段过渡/铺垫把新发展坐实（不强行改回前文）；手动模式：交用户选「改冲突（选基准）」或「加设定（补衔接）」 |

### 3.2 修复执行（逐文件写回）

#### 3.2-A 自动模式（选 [A] 时）

对每一处选中的冲突，按固定顺序处理：**定位 → 套用 3.1 规则判定权威方 → 改写对齐 → 单文件落地 → 记录**。不打断用户。

> **自动模式对「疑似新增设定」的默认行为**：当一处章间情节差异被判定为疑似新增设定（后章新发展仅缺过渡），自动模式**不强行把后章改回前文**，而是**自动新增相关设定/过渡**——在前文或过渡处补一段合理的铺垫或转折戏，把后章的新发展坐实为连贯的剧情推进（例如第1章敌对、第2章联手，则在第1章末或第2章开头补一句/一段"某事件让两人暂搁前嫌"的衔接）。这样既保留作者可能想要的设定，又消除"情节断层"的阅读违和。

> 自动模式修复示例：
> - 冲突（人设"谨慎" vs 第04章主角当众挑衅）：权威方=人设 → 改写第04章，让主角以更隐忍/迂回方式达成目的，保留冲突张力但不崩人设。
> - 冲突（时间线矛盾"三日后出发" vs "昨日抵达"）：以后文为准 → 修正前文，把"三日后出发"改为"当日兼程出发，次日清晨便抵"。
> - 冲突（悬念钩子不符：大纲"密信" vs 正文"安然入睡"）：权威方=大纲 → 改写章尾，落到"信使带来了密信"的钩子上。

#### 3.2-B 手动逐条模式（选 [B] 时）— 每处冲突都提供选项

按冲突编号从小到大，对**每一处**冲突单独处理，**不合并、不批量默认**：

1. **呈现该冲突**：用文字说明「编号 / 维度 / 位置 / 双方表述 / 候选改法（来自下表）」。
2. **给出可选改法（AskUserQuestion）**：根据该冲突类型，从下方「各类型候选改法表」取出可行的 2~3 种方案作为选项，请用户**选择其中一种作为基准**（满足要求 1：多种修改可能 → 用户选一种）：
   - 选项即各候选方案（如「改正文以符合人设」「改人设以符合正文」「两者协调」）；
   - 对**章间情节矛盾**类冲突，额外提供 **[视为新增设定：保留新发展并补过渡衔接]** 选项——让用户明确选择"这是冲突要改"还是"这是新设定要保留并补衔接"；
   - 额外提供 **[我自己指定改法]** 选项，让用户用自然语言描述想要的改法；
   - 额外提供 **[对剩余同类冲突套用此方案]** 选项（可选快捷项，用于大量同质冲突时加速；它仍经由用户本次主动选择，不剥夺逐条选择权）。
3. **仅一种合理改法时也要询问**：如姓名错写唯一种族正确名可归，选项简化为「按唯一正确改法修复 / 我自己指定」，确保每处都经用户确认。
4. **落地**：按用户所选方案改写对齐，**仅写回涉及的那一个文件**；记录「编号、所选方案、涉及文件」。
5. 处理**下一处**冲突（回到步骤 1），直到全部处理完。

> **多冲突要求（要求 2）**：手动模式对每一处冲突都独立弹一次 AskUserQuestion，确保"多个冲突，每个都提供选项令用户选择"，不会一次性默认处理多起冲突。

**各类型候选改法表**（手动模式据此生成每处冲突的选项）：

| 冲突类型 | 候选改法（供用户选一种作为基准） |
|----------|----------------------------------|
| 维度一·人物设定 ↔ 正文 | ① 改正文以符合人物设定（默认）② 改人物设定以符合正文 ③ 两者协调（微调正文＋微调人设） |
| 维度二·大纲情节 ↔ 正文 | ① 改正文回归大纲（默认）② 改大纲贴合正文 ③ 两者协调 |
| 维度三·上文 ↔ 下文（含章间情节矛盾） | ① 改前文（第X章）对齐后文（第X+1章）② 改后文（第X+1章）对齐前文 ③ 两者协调（前后都微调，补转折）④ **视为新增设定**：保留后章新发展，仅补一段过渡/铺垫把其坐实（用于"疑似新增设定"） |
| 维度四·大纲 / 人设内部（Timing 1） | 以「保留哪一版」为选项（保留用户新改版 / 保留旧版）；姓名类按「异名归一」归正到正确角色 |
| 姓名 / 称谓冲突 | ① 按档案真名归正（默认）② 归正到用户指定的另一角色 ③ 我自己指定 |

> 说明：当章间情节差异被判定为「疑似新增设定」时，**自动模式默认采用 ④（补设定/过渡）**；**手动模式把 ④ 也列为选项**，由用户决定「改冲突（①②③）」还是「加设定（④）」。

> 手动模式示例（某处「人设'谨慎' vs 第04章主角当众挑衅」）：
> AskUserQuestion 呈现：冲突[1]·维度一·第04章，人设"谨慎"与正文主角当众挑衅守卫矛盾。
> 选项：[改正文以符合人设（推荐）] [改人设以符合正文] [两者协调] [我自己指定改法]
> 用户选「改正文以符合人设」→ 改写第04章让主角以更隐忍/迂回方式达成目的 → 仅写回第04章。

> 自动 / 手动共用的改写对齐细则：
> - 改**正文**：仅写回当前这一章的 `.md`，用符合人设 / 大纲 / 前后文的方式重写矛盾处，保持"展示而非讲述"、不破坏悬念钩子与人物弧光。
> - 改**大纲/人设 JSON**：直接编辑对应 JSON 字段，使其与正文（当正文为权威）或彼此一致（Timing 1 内部不一致）。
> - 改**前文以对齐后文**：修正较早章节的表述，使时间线 / 因果 / 状态前后贯通。
> - 单文件落地：一次只写回一个文件。若同一处冲突横跨两章（如前文状态与后文不符），**只在需要修正的那一章修改**，另一章保持不变。

### 3.3 多轮复检（冲突可能出现多次）

人物 / 情节冲突**往往不止一处，且修复一处可能牵动别处**。因此：

1. 本轮所有选中冲突修复完成后，**重新执行一次「一、冲突检测」**，基于修改后的材料复扫。
2. 若复检仍发现冲突（含修复引入的新不一致）：
   - 生成**更新后的冲突报告**，回到「二、冲突呈现与用户选择」请用户决定下一轮是否继续修复。
   - 如此循环，直到**无冲突剩余**或**用户选择跳过结束**。
3. 若复检无冲突 → 进入第四步收尾。

> 设计为可多轮，正是为了应对"人物情节可能有多处冲突、需要修改多次"的情况。每一轮都尊重用户的选择权，不会在用户未确认时无限改动。

## 四、修复后检查

对每一处修改完成后的材料执行：

1. **字数检查**（仅当正文章节被改写时）：
   `python scripts/check_chapter_wordcount.py <章节文件路径>`
   - 若删除后字数低于 2000 → 用 [content-expansion.md](../guides/content-expansion.md) 补充**新**内容，且新内容不得引入新冲突。
2. **一致性复核**：修改后的段落与前后文是否自然衔接，黄金法则（展示而非讲述 / 冲突驱动 / 悬念钩子）是否仍满足。
3. **无新冲突检查**：确保修改没有意外制造人物 / 时间线 / 设定上的新矛盾。
4. **不破坏结构**：章节结尾悬念钩子、大纲 `keyEvents` 覆盖度、人物弧光不被削弱。

## 六、避坑指南

### 不要过度修改
- 保留角色标志性特征（口头禅、习惯动作）与必要承接（1~2 句足够），这是辨识度与连贯性的来源。
- 不要为消除一处小矛盾而重写整章；精准定位、最小改动。

### 不要破坏悬念与弧光
- 章节结尾钩子即使与上文有微调冲突，优先改前文承接而非删结尾钩子。
- 修复人设矛盾时，保留该章的冲突 / 转折力度（黄金法则二）。

### 不要引入新问题
- 改写后的措辞不能比原文差（抽象程度、细节密度）。
- 跨章节引用保持一致（角色姓名、地点名、设定名词不改错）。
- 改 JSON 字段后，确认 `01-大纲.json` 与 `02-写作计划.json` 的章节 `title` / `chapterNumber` 仍对应。

### 尊重用户意图
- Timing 1 用户主动改过的设定即视为用户最终意图，不要擅自"还原"回旧设定。
- 任何无法判定权威方的冲突，交回用户决策，不替用户拍板。
