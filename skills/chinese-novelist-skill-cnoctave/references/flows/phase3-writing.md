# 第三阶段：疯狂创作

**重要：全程无需再次向用户确认，必须以批次并发的方式完成全部章节的创作**

> **并行原则（贯穿所有模式）**：章节创作以【批次】为单位并行推进，而非逐章串行等待。
> - 子Agent模式 [B] 与 Agent Teams模式 [C]：在【同一条消息】中一次性并发派发本批所有章节的子 Agent，多个章节同时写作；本批全部返回并校验后，再派发下一批。
> - 主 Agent 串行模式 [A]：单个 Agent 无法同时写多章，仍逐章写，仅作为无子 Agent 环境或短篇的轻量回退。
>
> 并行安全由两条机制保证：**① 大纲前置规划**（每章规划信息在 Phase 2 已全部写好，是独立创作的完整依据，不依赖其他章的动态摘要）；**② 每章独立状态文件**（并发写作期间互不修改共享 JSON，由主 Agent 在批末统一合并）。详见第 5 节。

---

## 0. 启动检测与模式读取

开始创作前：
1. 读取 `02-写作计划.json`
2. 读取 `writingMode` 字段，进入对应模式流程
3. 确保 `{projectPath}/状态/` 目录存在（用于存放每章独立状态文件）
4. 如果存在 `status: "in_progress"` 的章节（中断续写）→ 将其视为待补写的章，从断点继续
5. 如果所有章节 `status: "pending"` → 从第 1 章开始
6. 如果存在 `status: "failed"` 的章节（Phase 4 回退）→ 加入重试批次
7. 设定并行批次大小 `parallelBatchSize`（默认 `5`；可在模式选择时由用户指定，或设为 `"all"` 一次性并发全部章节）

## 1. 逐章创作流程（通用，所有模式共用）

每章创作时严格执行以下步骤：

### 步骤 1: 写前分析（必须执行）

1. 读取 `02-写作计划.json` — 查看各章节状态，确定下一个待创作章节
2. **读取 `01-大纲.json`** — 找到当前章节的规划信息，提取：核心事件、承接上章、悬念钩子、出场人物、场景列表
2.5. **读取 `00-人物档案.json`** — 根据大纲中本章的「出场人物」列表，提取每个出场角色的：性格核心、致命缺陷、说话风格/口头禅、恐惧/弱项、与其他角色的关系
3. 标记本章为「进行中」：
   - **并行模式 [B]/[C]**：在**本章独立状态文件** `状态/第XX章-state.json` 中写入 `status: "in_progress"`（**不要**在此阶段修改共享的 `02-写作计划.json`）
   - **串行模式 [A]**：将 `02-写作计划.json` 中本章 `status` 设为 `"in_progress"`

### 步骤 2: 撰写

4. 创建章节文件 — 文件名格式：`第{XX}章-{章节标题}.md`（标题来自 `02-写作计划.json` 中的 `title` 字段），使用 [chapter-template.md](../guides/chapter-template.md) 模板
5. **基于大纲规划创作** — 严格按照大纲中本章的核心事件和场景列表撰写正文（大纲已包含承接上章与悬念钩子的完整规划，是独立创作的充分依据）
5.5. **撰写章首引子** — 按大纲中本章的章首引子类型，参考 [hook-techniques.md](../guides/hook-techniques.md)「章首引子七式」，创作 50-150 字的引子文字
6. 撰写正文 — **每章必须达到 3000-5000 字**
   - 章首引子：已创作（步骤 5.5）
   - 正文开头：第一段必须使用 [chapter-guide.md](../guides/chapter-guide.md) 十种开头技巧之一，建立即时冲突
   - 张力节奏：全章至少 2 个张力波峰，连续 500 字以上无冲突时必须引入新张力
   - 对话要求：每章至少 30% 对话内容，每段对话必须有潜台词或推进情节目的
   - 意外转折：每章至少 1 个读者预期之外的事件或信息
   - 人物一致性：对话和行为必须严格符合步骤 2.5 提取的角色设定
   - 内容不足？使用 [content-expansion.md](../guides/content-expansion.md) 扩充技巧
7. 设置结尾钩子 — 按大纲中本章的悬念钩子设计 → [hook-techniques.md](../guides/hook-techniques.md)「悬念钩子十三式」
8. **字数检查** — 必须使用脚本检查：`python scripts/check_chapter_wordcount.py <章节文件路径>`

### 步骤 3: 撰写后优化

9. 连贯性检查 — 人物一致性、情节连贯、节奏控制
9.5. **张力检查** — 检查全章节奏是否有波峰波谷、对话是否有个性、是否有意外转折
10. **深度润色（去除AI味）** — 重点检查并修改：
    - **去除过度修饰的形容词**：删减"璀璨"、"瑰丽"等AI常用词堆砌
    - **减少抽象陈述**：把"心中涌起复杂的情感"改为具体动作/对话
    - **打破四字格律**：避免"心潮澎湃、热血沸腾"等陈词滥调
    - **增加口语化表达**：人物对话要有个性
    - **优化节奏感**：长句短句交替
    - **细节具象化**：用具体细节替代笼统描述
11. **字数检查** — 再次使用脚本确认

### 步骤 4: 收尾

12. 生成章节摘要（300-500字，供后续增强阶段作连贯性参考）：
    - **并行模式 [B]/[C]**：将摘要写入**本章独立状态文件** `状态/第XX章-state.json` 的 `summary` 字段（**不要**在此阶段修改共享的 `01-大纲.json`）
    - **串行模式 [A]**：在 `01-大纲.json` 的章节摘要区追加
13. 标记本章完成：
    - **并行模式 [B]/[C]**：将独立状态文件中本章 `status` 设为 `"completed"`，填入 `wordCount`、`wordCountPass`
    - **串行模式 [A]**：更新 `02-写作计划.json` — 将本章 `status` 设为 `"completed"`，填入 `wordCount`

## 2. 串行模式（writingMode: "serial"）

**主 Agent 自己逐章创作，全程不中断。每写完一章立即校验，通过后再写下一章。仅作为无子 Agent 环境或短篇小说的轻量回退选项。**

### 逐章主循环（serial）

```
FOR 每一章（按 02-写作计划.json 章节顺序，一次只处理一章）:
    1. 执行「逐章创作流程」（步骤 1-4）写完【当前这一章】
    2. 立即进入第四阶段，对【当前这一章】执行校验与修复
       - 字数不达标 → 当场重写当前章（最多3轮），通过后再继续
    3. 当前章确认完成 → 读取 JSON 认领下一章
    ⚠️ 完成一章并校验通过后，才处理下一章，不要向用户确认，不要停下来
所有章节完成 → 进入第4.5步：可选增强阶段
```

**关键提醒（每章完成后必须遵守）**：
> 本章已完成并通过校验。立即读取 `02-写作计划.json`，认领下一个 pending 章节，开始下一章创作。不要使用 AskUserQuestion，不要向用户确认，不要停下来。你必须把所有章节创作完成才能与用户报告。

## 3. 子Agent并行模式（writingMode: "subagent-parallel"）★ 重构重点

**核心机制**：主 Agent 将待写章节按 `parallelBatchSize` 分批，在【同一条消息】中并发派发本批所有章节的子 Agent，**多个章节同时写作**；本批全部返回后由主 Agent 统一合并状态并批量校验，再派发下一批。这是中长篇创作的默认推荐模式。

### 主 Agent 流程

```
1. 读取 02-写作计划.json，收集所有 pending / failed / in_progress(中断) 章节
2. 读取 parallelBatchSize（默认 5，或全部）
3. WHILE 仍有未完成的章节（pending 或 failed 且 retryCount < 3）:
   a. 取本批 = 接下来至多 parallelBatchSize 个未完成章节
   b. 【同一条消息】并发派发本批每个章节一个子 Agent（每个子 Agent 只写一章 + 写独立状态文件）
   c. 等待本批所有子 Agent 返回
   d. 读取本批每个 状态/第XX章-state.json，合并进度到 02-写作计划.json：
      - 缺状态文件 / 子 Agent 报错 → 该章标记 failed，下批重试
      - status=completed → 填入 wordCount、wordCountPass
   e. 对本批各章【并发】执行第四阶段校验（见 phase4-validation.md 批量校验路径）
      - 校验失败章（wordCountPass=false / failed）计入下一批重试，retryCount+1
   f. 全部通过 → 进入下一批；达到 3 次仍不通过的章保留记录但不阻塞
4. 所有章节 completed → 将各独立状态文件中的 summary 合并回 01-大纲.json 对应章节
5. 进入第4.5步：可选增强阶段
```

> 第 b 步是关键：必须在一个消息里同时发出本批所有子 Agent 的派发调用，才能形成真正的并发。逐次派发（等一个回来再发下一个）会退化为串行。

```
5. 正文开头第一段必须使用 chapter-guide.md 十种开头技巧之一
6. 全章至少 2 个张力波峰，连续 500 字以上无冲突必须引入新张力
7. 每章至少 30% 对话，对话必须有潜台词和角色个性，**对话风格必须符合人物档案中的设定**
```

**并发安全（为什么不会冲突）**：
- 每个子 Agent 只写自己的章节文件和自己的 `状态/第XX章-state.json`，文件互不重叠。
- 并行期间**没有任何子 Agent 修改**共享的 `02-写作计划.json` 与 `01-大纲.json`。
- 所有共享文件的写入由主 Agent 在批末统一合并完成，从根本上消除并发写冲突。

## 4. Agent Teams 并行模式（writingMode: "agent-teams"）

**Claude Code 特有模式**。需用户手动开启。通过 TeamCreate 创建写作团队，多个成员**同时**认领并创作不同章节，实现批次并发协作。

### 主 Agent 流程

```
1. 使用 TeamCreate 创建写作团队
2. 派生 3-5 个团队成员（使用 Agent 工具 + team_name 参数）
3. 使用 TaskCreate 为每一章创建一个任务（一章 = 一个任务）
4. 按 parallelBatchSize 分批，将本批任务分发给多个成员，成员各自【并发】认领并写不同章节
```

## 5. 并行安全与连续性机制（核心新增）

### 5.1 大纲前置规划 = 并行的连续性保障

Phase 2 已为每章写好完整的、互相独立的规划：`coreEvent`（核心事件）、`continuationFromPrevious`（承接上章）、`cliffhanger`（悬念钩子）、`characters`（出场人物）、`scenes`（场景列表）、`keyEvents`（关键词）。**这些是每章独立创作的充分依据**，因此并行写作时不依赖其他章节的动态摘要即可保证剧情衔接。

- 串行模式下动态追加的 `summary` 摘要，在并行模式下不作为创作前置依赖；
- 并行模式把 `summary` 暂存于独立状态文件，待全部完成后由主 Agent 合并回 `01-大纲.json`，供后续增强阶段作连贯性参考。

### 5.2 每章独立状态文件

- 路径：`{projectPath}/状态/第{XX}章-state.json`
- 结构：
  ```json
  {
    "chapterNumber": 1,
    "title": "章节标题",
    "filePath": "第01章-章节标题.md",
    "status": "completed",
    "wordCount": 3570,
    "wordCountPass": true,
    "summary": "300-500字摘要……",
    "retryCount": 0
  }
  ```
- 并发写作期间，子 Agent 只写自己的状态文件与章节文件，**绝不修改** `02-写作计划.json` / `01-大纲.json`。
- 批末由主 Agent 读取所有独立状态文件，统一合并回 `02-写作计划.json` 与 `01-大纲.json`，消除并发写冲突。

### 5.3 批次合并与重试

- 每批子 Agent 全部返回后，主 Agent 合并本批状态文件；缺失状态文件或子 Agent 报错的章标记 `failed`，进入下一批重试。
- 单章最多重试 3 轮（retryCount 计入状态文件）；达到 3 次仍不通过的章保留记录、标注但不阻塞其他章节。
